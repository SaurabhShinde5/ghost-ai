## Architecture Decisions

- Route protection lives in `proxy.ts` at the project root, not `middleware.ts` — Next.js 16 renamed Middleware to Proxy. `clerkMiddleware` is protected-by-default: only the sign-in/sign-up URL env vars are public.
- Clerk appearance is themed via `dark` from `@clerk/ui/themes` (not `@clerk/themes`) with all `variables` pointing at the app's existing CSS custom properties, so Clerk UI inherits the design tokens with zero hardcoded colors.
- Public-route config keys off Clerk's standard `NEXT_PUBLIC_CLERK_SIGN_IN_URL` / `NEXT_PUBLIC_CLERK_SIGN_UP_URL` env vars rather than string literals, so auth paths stay single-sourced.
- Dark-only theme: shadcn's light `:root` / dark `.dark` split is collapsed into a single dark palette in `globals.css` (values from `ui-context.md`), and `<html>` carries the `dark` class so shadcn `dark:` variants stay active. No light mode.
- Project UI mock data lives in `lib/mock-projects.ts` (not the legacy `data/` directory, which `architecture-context.md` marks unused for new artifacts); the shared `Project` type lives in `types/project.ts`. Both are placeholders until Prisma-backed project CRUD lands.
- Dialog/form/loading state plus the create/rename/delete mutations are centralized in `hooks/use-project-actions.ts` (formerly the mock `use-project-dialogs.ts`). The `/editor` page is a thin server component that fetches data and delegates all interactivity to `components/editor/editor-shell.tsx`, keeping the server/client boundary clean while composing home + sidebar + dialogs against one state source.
- `getUserProjects()` (`lib/projects.ts`) is the single server-side read path for the editor home, sharing the owned-projects query shape with `GET /api/projects`. Shared projects are resolved by matching the Clerk user's email addresses (via `currentUser()`) against `ProjectCollaborator.email`, since collaborators are keyed by email, not Clerk user ID.
- Project id / Liveblocks room id alignment: the create flow generates `roomId = slugify(name)-<suffix>` on the client and sends it as the project `id`; `POST /api/projects` uses a client-provided `id` when present (else the cuid default). This keeps `project.id === roomId` so the workspace route and canvas room resolve from one identifier.
- Prisma v7 with the `prisma-client` generator (ESM output to `app/generated/prisma`, gitignored) and required driver adapters. The datasource URL is single-sourced through `prisma.config.ts` (`datasource.url = process.env.DATABASE_URL`) rather than in `schema.prisma`. `lib/prisma.ts` picks the transport by URL protocol so a `prisma+postgres://` (Accelerate) or a direct `postgres://` (pg adapter) URL both work with no code change. Schema is split multi-file: connector/generator in `prisma/schema.prisma`, models in `prisma/models/*.prisma`.
- Project mutation routes (`PATCH`/`DELETE`) are owner-only per `06-project-apis.md` (non-owner → `403`), intentionally stricter than `architecture-context.md`'s general "owner or collaborator can mutate" rule. Collaborator write access can be reintroduced when the collaborator feature is actually built.
- `lib/prisma.ts` normalises `createPrismaClient` to a single `PrismaClient` return type (Accelerate branch cast to it). The prior union of the two transport client types was fine until a real model query was type-checked, at which point member signatures became incompatible; pinning the type is the root-cause fix and keeps both transports interchangeable at the call site.
- Server-side access control for the workspace lives in `lib/project-access.ts`, separate from the read-only `getUserProjects()` in `lib/projects.ts`. The `/editor/[roomId]` page composes `getCurrentIdentity()` + `getAccessibleProject()` (owner-or-collaborator check) and renders `AccessDenied` for both the not-found and no-access cases — the page never leaks whether a project exists. Missing/unauthorized deliberately collapse to one outcome per `08-editor-workspace-shell.md`.
- Collaborator access checks match against the identity's full verified-email list (`ClerkIdentity.emails`), not only the primary address, so the workspace access decision stays consistent with `getUserProjects()`'s shared-project resolution. The spec's "userId + primary email" identity is provided (`userId`, `primaryEmail`) with `emails` added for that consistency.
- The `/editor/[roomId]` workspace uses its own `WorkspaceNavbar` + `WorkspaceShell` rather than reusing the editor-home `EditorNavbar`/`EditorShell`: the workspace navbar carries the project name, Share action, and AI toggle, which the home navbar has no concept of. Both shells share `ProjectSidebar`, `useProjectActions`, and `ProjectDialogs`, so create/rename/delete behave identically across home and workspace. The active room is highlighted via a new optional `activeProjectId` prop on `ProjectSidebar`.
- `PageProps<"/editor/[roomId]">` (Next.js 16's generated route-typing helper, `params` async) types the dynamic page. These types are emitted by `next dev`/`next build`/`next typegen`; run `next typegen` before a bare `tsc --noEmit` on a fresh checkout or the route string won't satisfy `AppRoutes`.
- Collaborators are keyed by email with no local user table (per `09-share-dialog.md`). Clerk profile data (name, avatar) is resolved on demand server-side via `enrichCollaborators` (`clerkClient().users.getUserList({ emailAddress })`), matching each Clerk user's emails back to the stored collaborator emails; any missing user or Clerk failure degrades to email-only rendering. Collaborator mutations (invite/remove) are owner-only server-side — consistent with the project rename/delete routes and stricter than `architecture-context.md`'s general collaborator-mutation rule; listing is allowed for any member (`hasProjectAccess`). Removal is keyed by `ProjectCollaborator.id` scoped to the project (`deleteMany({ id, projectId })`) rather than by email in the URL path.
- Design tokens from `ui-context.md` are exposed as Tailwind utilities via `@theme inline` (e.g. `bg-base`, `bg-surface`, `text-copy-*`, `text-brand`, `bg-accent-dim`, `text-ai`), layered alongside shadcn's semantic tokens.
- Liveblocks auth uses **access tokens** (`prepareSession` + `session.allow(roomId, FULL_ACCESS)`), not ID tokens: we already verify membership server-side via `getAccessibleProject`, and each token grants access to exactly one room (the project), so per-room `usersAccesses` bookkeeping is unnecessary and the spec explicitly asks for a "session token". Rooms are created lazily via `getOrCreateRoom` at auth time (`defaultAccesses: []` → private; access flows only through the session token). The Liveblocks room ID **is** the project ID, keeping room ⇄ project resolution single-sourced (consistent with the create flow's `project.id === roomId`).
- `lib/liveblocks.ts` mirrors `lib/prisma.ts`: a globalThis-cached singleton (hot-reload-safe outside production) that throws when `LIVEBLOCKS_SECRET_KEY` is absent. Cursor colour is derived deterministically from the Clerk user ID (`getUserColor`) rather than stored, so it is stable and requires no schema.

- `CanvasNodeShape` is the six-shape ui-context set (`rectangle | diamond | circle | pill | cylinder | hexagon`), and `NODE_SHAPES`/`NODE_COLORS`/`DEFAULT_NODE_COLOR` now live in `types/canvas.ts` as ui-context documents — the earlier 3-value placeholder (`rectangle | ellipse | diamond`) from `11-base-canvas` was a stub. New nodes are created by feeding a React Flow `{ type: "add", item }` change to `useLiveblocksFlow`'s `onNodesChange` (the hook exposes no `setNodes`), which routes through the same Storage-sync path as every other change. Node size travels as top-level `width`/`height` (v12 sizes the wrapper from them and Liveblocks syncs them; only `measured` is excluded from sync). Drag payloads move over a dedicated `SHAPE_DRAG_MIME` dataTransfer type so unrelated drops are ignored, and `screenToFlowPosition` (needs `ReactFlowProvider`, hence `Canvas` now wraps an inner component) maps the drop point into canvas coordinates.

- The collaborative canvas is split into two client components: `CanvasRoom` owns the Liveblocks room lifecycle (provider + `RoomProvider` + suspense/error boundaries) while `Canvas` owns only the React Flow ↔ Storage binding via `useLiveblocksFlow`. This keeps the room/connection concern separate from the flow-rendering concern so cursors, controls, and custom rendering can be added to `Canvas` without touching room setup. Connection failures are handled with `useErrorListener` (`ROOM_CONNECTION_ERROR` → in-place error fallback) rather than adding a `react-error-boundary` dependency — connection errors surface through the listener, and `ClientSideSuspense` already covers the loading state. The Liveblocks `Storage` global type stays `Record<string, never>`; `useLiveblocksFlow` manages its own `"flow"` storage key internally and does not require the root Storage type to declare it. `CanvasNodeData`/`CanvasEdgeData` are declared with `interface … extends Record<string, unknown>` (rather than a plain `interface`) specifically because React Flow's `Node`/`Edge` data generics require assignability to `Record<string, unknown>`, which a bare interface (no index signature) does not satisfy.
