- `fix-canvas-save-concurrency` — replaced the canvas save's blind blob overwrite with optimistic concurrency so stale retries / slower concurrent autosaves can't clobber a newer snapshot. **Route** (`app/api/projects/[projectId]/canvas/route.ts` `PUT`): reads an optional `If-Match` request header and, when present, passes it as `put({ ifMatch })` (keeping `allowOverwrite: true`); a `BlobPreconditionFailedError` (ETag mismatch) is caught and returned as `409 { error }` instead of overwriting; the response now returns `{ url, etag }` plus a standard `ETag` header so the caller can guard its next save. The first save (no known ETag) still does an unconditional overwrite, so behavior is backward-compatible. **Client** (`hooks/use-canvas-autosave.ts`, the sole caller — a server-only change would be inert since optimistic concurrency needs the writer to declare its base version): tracks the last persisted ETag in `etagRef`, sends it as `If-Match`, captures the returned `etag` on success, and on `409` drops the stale ETag and retries unconditionally within the existing bounded retry loop so the current (shared, authoritative Liveblocks) state still persists. Verified with `@vercel/blob@2.5.0` types (`CommonCreateBlobOptions.ifMatch`, `PutBlobResult.etag`, exported `BlobPreconditionFailedError`). Scope: change confined to the canvas save flow (route handler + its caller); `GET`/load path untouched. Verified with `eslint` + `npm run build`.

- `21-canvas-autosave` — collaborative canvas persistence via Vercel Blob (Prisma stores only the blob URL). **Schema:** reused the existing `Project.canvasJsonPath String?` field (already present from `05-prisma`) — no migration needed. **API:** new `app/api/projects/[projectId]/canvas/route.ts` — `PUT` validates the body into a `CanvasSnapshot` (`{ nodes, edges }`, both arrays), uploads it to Vercel Blob at the stable path `canvas/{projectId}.json` (`@vercel/blob` `put` with **`access: "private"`** — the store is configured private — `contentType: "application/json"`, `addRandomSuffix: false`, `allowOverwrite: true` so each save overwrites), then stores `blob.url` on the project via `prisma.project.update`; `GET` reads `project.canvasJsonPath` (returns `{ canvas: null }` when unset), otherwise reads the blob back **through the SDK `get(url, { access: "private" })`** (a private blob URL isn't publicly fetchable, so a plain `fetch` won't work — `get` authenticates with the read-write token), streams it to JSON, and returns `{ canvas }`. Both the `put`+update and the `get` are wrapped in try/catch (`PUT` → predictable `500 { error }`, `GET` → `{ canvas: null }`). Both handlers `getCurrentIdentity()` → `401`, then `getAccessibleProject(projectId, identity)` → `404` when null (owner **or** collaborator may save/load, matching the collaboration model — stricter owner-only rules are only for rename/delete/invite). **Autosave hook:** new `hooks/use-canvas-autosave.ts` (`"use client"`) — `useCanvasAutosave({ projectId, nodes, edges })` returns a `CanvasSaveStatus` (`idle`/`saving`/`saved`/`error`). It serializes the graph with volatile view fields stripped (`selected`/`dragging`/`measured` off nodes, `selected` off edges) so selection/hover/measure churn never counts as a change; the **first render baselines** the loaded state (records `lastSaved`, no write) so a save only fires once the graph actually changes; changes flip status to `saving`, then a `DEBOUNCE_MS = 1000` `setTimeout` `PUT`s the serialized snapshot → `saved` on ok, `error` on failure. **Cancellation** cancels only the pending debounce timer (`clearTimeout`) and marks the run `cancelled` so a stale in-flight request's result is ignored — it deliberately does **not** `AbortController.abort()` the fetch: aborting mid-flight surfaced an `AbortError` in Next dev and, under rapid post-edit churn (React Flow's measured/position updates), starved the write by cancelling every in-flight save so the blob never persisted. Letting the request finish (ignoring stale results, with the newer change scheduling the next save) means the latest state always converges to the blob. **Transient-failure resilience:** the save runs a bounded retry loop (`MAX_ATTEMPTS = 3`, linear `RETRY_BACKOFF_MS = 800` backoff, cancellation-aware between attempts) so a one-off `TypeError: Failed to fetch` (dropped/reset connection, dev-server recompile, or a browser extension intercepting `fetch`) is retried before the indicator flips to `error`, rather than permanently failing until the next edit. The `PUT` route wraps the Vercel Blob `put` + `prisma.project.update` in a try/catch that logs and returns a predictable `500 { error }` instead of throwing. **Load:** `CanvasInner` (`canvas.tsx`) runs a **once-per-room** effect (deps `[projectId]`) that reads the mount-time room state + sync handlers from a `useRef` snapshot (`loadContext`) so it never lists `nodes`/`edges` in its deps — which it mutates — and therefore can't self-cancel. It **skips entirely when the room already has nodes or edges** (never overwrites active collaboration); for an empty room it `GET`s the canvas, and when `data.canvas` is present adds its nodes then edges through the same `onNodesChange`/`onEdgesChange` `add`-change Storage-sync path as templates, then `setTimeout(80ms)` → `reactFlow.fitView`. It uses a plain `cancelled` flag (no `AbortController`) so unmount is handled without surfacing a spurious `AbortError`. Loading into Storage triggers the autosave hook's first real change (re-persisting the loaded state), which is harmless. **Save indicator:** new `components/editor/save-status.tsx` — a non-interactive (disabled, `disabled:opacity-100`) ghost `Button` styled as the editor's Save button, mapping status → icon+label+token color (`Check`/"Saved"/`text-copy-muted|secondary`, spinning `Loader2`/"Saving…", `CircleAlert`/"Save failed"/`text-error`), `aria-live="polite"`. **Wiring:** autosave status is lifted to `WorkspaceShell` (`saveStatus` state) via a new `onSaveStatusChange` callback threaded `WorkspaceShell` → `CanvasRoom` (passes `projectId={roomId}`) → `Canvas` → `CanvasInner` (a `useEffect` reports the hook's status up); `WorkspaceNavbar` renders `<SaveStatus>` before the Templates button. Installed `@vercel/blob`; `BLOB_READ_WRITE_TOKEN` was already present in `.env.local`. `CanvasSnapshot` added to `types/canvas.ts`. Scope honored: metadata stays in Prisma, canvas JSON in Blob, autosave debounced, editor shows save status, saved canvas skips loading when the room is non-empty. Verified with `npm run build` (`/api/projects/[projectId]/canvas` server-rendered on demand) and `eslint`.

## Architecture Decisions

- Route protection lives in `proxy.ts` at the project root, not `middleware.ts` — Next.js 16 renamed Middleware to Proxy. `clerkMiddleware` is protected-by-default: only the sign-in/sign-up URL env vars are public.
- Clerk appearance is themed via `dark` from `@clerk/ui/themes` (not `@clerk/themes`) with all `variables` pointing at the app's existing CSS custom properties, so Clerk UI inherits the design tokens with zero hardcoded colors.
- Public-route config keys off Clerk's standard `NEXT_PUBLIC_CLERK_SIGN_IN_URL` / `NEXT_PUBLIC_CLERK_SIGN_UP_URL` env vars rather than string literals, so auth paths stay single-sourced.
- Dark-only theme: shadcn's light `:root` / dark `.dark` split is collapsed into a single dark palette in `globals.css` (values from `ui-context.md`), and `<html>` carries the `dark` class so shadcn `dark:` variants stay active. No light mode.
- Project UI mock data lives in `lib/mock-projects.ts` (not the legacy `data/` directory, which `architecture-context.md` marks unused for new artifacts); the shared `Project` type lives in `types/project.ts`. Both are placeholders until Prisma-backed project CRUD lands.
- Dialog/form/loading state plus the create/rename/delete mutations are centralized in `hooks/use-project-actions.ts` (formerly the mock `use-project-dialogs.ts`). The `/editor` page is a thin server component that fetches data and delegates all interactivity to `components/editor/editor-shell.tsx`, keeping the server/client boundary clean while composing home + sidebar + dialogs against one state source.
- `getUserProjects()` (`lib/projects.ts`) is the single server-side read path for the editor home, sharing the owned-projects query shape with `GET /api/projects`. Shared projects are resolved by matching the Clerk user's email addresses (via `currentUser()`) against `ProjectCollaborator.email`, since collaborators are keyed by email, not Clerk user ID.
- Project id / Liveblocks room id alignment: the create flow generates `roomId = slugify(name)-<suffix>` on the client and sends it as the project `id`; `POST /api/projects` uses a client-provided `id` when present (else the cuid default). This keeps `project.id === roomId` so the workspace route and canvas room resolve from one identifier.
- Prisma v7 with the `prisma-client` generator (ESM output to `app/generated/prisma`, gitignored) and required driver adapters. The datasource URL is single-sourced through `prisma.config.ts` (`datasource.url = process.env.DATABASE_URL`) rather than in `schema.prisma`. `lib/prisma.ts` picks the transport by URL protocol so a `prisma+postgres://` (Accelerate) or a direct `postgres://` (pg adapter) URL both work with no code change. Schema is split multi-file: connector/generator in `prisma/schema.prisma`, models in `prisma/models/*.prisma`.
- Project mutation routes (`PATCH`/`DELETE`) are owner-only per `06-project-apis.md` (non-owner → `403`), intentionally stricter than `architecture-context.md`'s general "owner or collaborator can mutate" rule. Collaborator write access can be reintroduced when the collaborator feature is actually built.
- `lib/prisma.ts` normalises `createPrismaClient` to a single `PrismaClient` return type (Accelerate branch cast to it). The prior union of the two transport client types was fine until a real model query was type-checked, at which point member signatures became incompatible; pinning the type is the root-cause fix and keeps both transports interchangeable at the call site.
- Server-side access control for the workspace lives in `lib/project-access.ts`, separate from the read-only `getUserProjects()` in `lib/projects.ts`. The `/editor/[roomId]` page composes `getCurrentIdentity()` + `getAccessibleProject()` (owner-or-collaborator check) and renders `AccessDenied` for both the not-found and no-access cases — the page never leaks whether a project exists. Missing/unauthorized deliberately collapse to one outcome per `08-editor-workspace-shell.md`.
- Collaborator access checks match against the identity's full verified-email list (`ClerkIdentity.emails`), not only the primary address, so the workspace access decision stays consistent with `getUserProjects()`'s shared-project resolution. The spec's "userId + primary email" identity is provided (`userId`, `primaryEmail`) with `emails` added for that consistency.
- The `/editor/[roomId]` workspace uses its own `WorkspaceNavbar` + `WorkspaceShell` rather than reusing the editor-home `EditorNavbar`/`EditorShell`: the workspace navbar carries the project name, Share action, and AI toggle, which the home navbar has no concept of. Both shells share `ProjectSidebar`, `useProjectActions`, and `ProjectDialogs`, so create/rename/delete behave identically across home and workspace. The active room is highlighted via a new optional `activeProjectId` prop on `ProjectSidebar`.
- `PageProps<"/editor/[roomId]">` (Next.js 16's generated route-typing helper, `params` async) types the dynamic page. These types are emitted by `next dev`/`next build`/`next typegen`; run `next typegen` before a bare `tsc --noEmit` on a fresh checkout or the route string won't satisfy `AppRoutes`.
- Collaborators are keyed by email with no local user table (per `09-share-dialog.md`). Clerk profile data (name, avatar) is resolved on demand server-side via `enrichCollaborators` (`clerkClient().users.getUserList({ emailAddress })`), matching each Clerk user's emails back to the stored collaborator emails; any missing user or Clerk failure degrades to email-only rendering. Collaborator mutations (invite/remove) are owner-only server-side — consistent with the project rename/delete routes and stricter than `architecture-context.md`'s general collaborator-mutation rule; listing is allowed for any member (`hasProjectAccess`). Removal is keyed by `ProjectCollaborator.id` scoped to the project (`deleteMany({ id, projectId })`) rather than by email in the URL path.
- Design tokens from `ui-context.md` are exposed as Tailwind utilities via `@theme inline` (e.g. `bg-base`, `bg-surface`, `text-copy-*`, `text-brand`, `bg-accent-dim`, `text-ai`), layered alongside shadcn's semantic tokens.
- Liveblocks auth uses **access tokens** (`prepareSession` + `session.allow(roomId, FULL_ACCESS)`), not ID tokens: we already verify membership server-side via `getAccessibleProject`, and each token grants access to exactly one room (the project), so per-room `usersAccesses` bookkeeping is unnecessary and the spec explicitly asks for a "session token". Rooms are created lazily via `getOrCreateRoom` at auth time (`defaultAccesses: []` → private; access flows only through the session token). The Liveblocks room ID **is** the project ID, keeping room ⇄ project resolution single-sourced (consistent with the create flow's `project.id === roomId`).
- `lib/liveblocks.ts` mirrors `lib/prisma.ts`: a globalThis-cached singleton (hot-reload-safe outside production) that throws when `LIVEBLOCKS_SECRET_KEY` is absent. Cursor colour is derived deterministically from the Clerk user ID (`getUserColor`) rather than stored, so it is stable and requires no schema.

- `CanvasNodeShape` is the six-shape ui-context set (`rectangle | diamond | circle | pill | cylinder | hexagon`), and `NODE_SHAPES`/`NODE_COLORS`/`DEFAULT_NODE_COLOR` now live in `types/canvas.ts` as ui-context documents — the earlier 3-value placeholder (`rectangle | ellipse | diamond`) from `11-base-canvas` was a stub. New nodes are created by feeding a React Flow `{ type: "add", item }` change to `useLiveblocksFlow`'s `onNodesChange` (the hook exposes no `setNodes`), which routes through the same Storage-sync path as every other change. Node size travels as top-level `width`/`height` (v12 sizes the wrapper from them and Liveblocks syncs them; only `measured` is excluded from sync). Drag payloads move over a dedicated `SHAPE_DRAG_MIME` dataTransfer type so unrelated drops are ignored, and `screenToFlowPosition` (needs `ReactFlowProvider`, hence `Canvas` now wraps an inner component) maps the drop point into canvas coordinates.

- The node color toolbar stores **only the fill** (`data.color`); the paired text color is never persisted but derived from the fill through the existing `nodeLabelTextColor`/`FILL_TO_TEXT` map. This keeps a single source of truth (one `data.color` write recolors both background and label) and means `CanvasNodeData` needs no new field. Recoloring reuses the exact `getNode` → `{ type: "replace" }` → `onNodesChange` path as label editing (`updateNodeColor` alongside `updateNodeLabel` on the `CanvasCallbacks` context), so it syncs through Liveblocks Storage with no server calls. The toolbar is a predefined-palette picker (`NODE_COLORS`), deliberately not a free-form color picker per the spec's scope limits.

- Node-data mutations from inside a custom node (label editing) go through `onNodesChange` with a `replace` change, **not** `useReactFlow().updateNodeData`/`updateNode`. `useLiveblocksFlow` renders `nodes` from Liveblocks Storage (via `useStorage`), so React Flow runs controlled — updating its internal store directly (what `updateNodeData` does) is not persisted and gets overwritten on the next Storage-driven render. Instead `CanvasInner` reads the current node with `getNode(id)`, spreads it, sets `data.label`, and dispatches `{ type: "replace", id, item }`; liveblocks-react-flow's `applyNodeChanges` reconciles it into Storage. Resize is already covered by the built-in `NodeResizer`, whose `dimensions` changes travel the same `onNodesChange` path. Custom nodes reach `updateNodeLabel` through a small `CanvasCallbacks` context (`canvas-context.ts`) rather than threading a non-serializable callback through synced node `data`.

- Custom edges reuse the same `onEdgesChange` + `{ type: "replace" }` sync path as node mutations, not `useReactFlow().updateEdge`/`updateEdgeData` — `useLiveblocksFlow` renders edges controlled from Liveblocks Storage, so writing React Flow's internal store directly is not persisted. `updateEdgeLabel` (on the `CanvasCallbacks` context alongside `updateNodeLabel`/`updateNodeColor`) reads the edge via `getEdge`, spreads it, sets `data.label`, and dispatches an `EdgeReplaceChange`; liveblocks-react-flow's `applyEdgeChanges` reconciles it into Storage. Four-side connectivity is achieved with four `type="source"` handles (unique ids per side) under the existing `ConnectionMode.Loose`, so any handle connects to any other without needing paired source+target handles per side. New connections are pushed to the custom renderer + arrow/light-stroke defaults via `defaultEdgeOptions` (not per-edge wiring). The custom edge separates the *visible* thin stroke (`BaseEdge`) from a wide invisible `transparent` interaction path that owns hover/dbl-click — this is how the edge becomes easy to hover/click without thickening the line. Edge labels are positioned with `getSmoothStepPath`'s returned `labelX`/`labelY` inside `EdgeLabelRenderer` (never a hand-computed midpoint), and the label input commits on blur/Enter/Escape (all three save, per spec). The light edge color is single-sourced as the `--edge-stroke` token (`#f8fafc` from ui-context) rather than a hardcoded hex at the call sites.

- The collaborative canvas is split into two client components: `CanvasRoom` owns the Liveblocks room lifecycle (provider + `RoomProvider` + suspense/error boundaries) while `Canvas` owns only the React Flow ↔ Storage binding via `useLiveblocksFlow`. This keeps the room/connection concern separate from the flow-rendering concern so cursors, controls, and custom rendering can be added to `Canvas` without touching room setup. Connection failures are handled with `useErrorListener` (`ROOM_CONNECTION_ERROR` → in-place error fallback) rather than adding a `react-error-boundary` dependency — connection errors surface through the listener, and `ClientSideSuspense` already covers the loading state. The Liveblocks `Storage` global type stays `Record<string, never>`; `useLiveblocksFlow` manages its own `"flow"` storage key internally and does not require the root Storage type to declare it. `CanvasNodeData`/`CanvasEdgeData` are declared with `interface … extends Record<string, unknown>` (rather than a plain `interface`) specifically because React Flow's `Node`/`Edge` data generics require assignability to `Record<string, unknown>`, which a bare interface (no index signature) does not satisfy.

- Canvas controls are split from keyboard handling: the visible bar (`CanvasControls`) is self-contained and re-reads `useUndo`/`useRedo`/`useCanUndo`/`useCanRedo` for its own disabled state, while `useKeyboardShortcuts` is a pure `window`-listener hook that receives the React Flow instance plus undo/redo handlers as arguments (no React Flow / Liveblocks imports of its own beyond the instance type), keeping it framework-agnostic and easy to test. `CanvasInner` calls `useUndo`/`useRedo` once and threads them into the hook. Zoom/fit use `{ duration: 200 }` for a smooth animation; the keyboard hook guards editable targets (`INPUT`/`TEXTAREA`/`SELECT`/`isContentEditable`) so shortcuts never fire mid-label-edit, and only claims `Cmd/Ctrl+Z/Y` — bare-modifier combos like browser zoom pass through. The bar sits bottom-left (`bottom-6 left-6`) to clear the bottom-center shape panel.

- Starter template import runs **inside** `CanvasInner`, not in `WorkspaceShell` — only there are the Liveblocks `onNodesChange`/`onEdgesChange` handlers and the React Flow `fitView` available. The navbar (outside the Liveblocks `RoomProvider`) only owns the modal's open flag (`isTemplatesOpen` in `WorkspaceShell`), which is threaded down `CanvasRoom` → `Canvas` → `CanvasInner`; the modal itself renders inside the canvas so `onImport` reaches the collaborative state. Import replaces (not appends): it emits `remove` changes for all current edges then nodes, then `add` changes for the template's nodes then edges — the same Storage-sync path as every other mutation, so no dedicated clear/setNodes API is needed. `fitView` is deferred with an 80ms `setTimeout` because `useLiveblocksFlow` renders nodes from Storage on the next tick and custom nodes must measure before a fit is meaningful. Template edges carry explicit `sourceHandle`/`targetHandle` ids (`bottom`/`top` or `right`/`left`) matching the node's four-side handles so routing is deterministic. The card diagram preview is a **standalone SVG** (`TemplatePreview`) that reimplements just enough shape drawing rather than mounting React Flow — deliberately lightweight per the spec, computing bounds from node positions/sizes and scaling to a fixed `260×150` viewport.

- The AI sidebar treats spec token names as ui-context roles, not literal Tailwind classes (the spec's `bg-accent`/`text-accent`/`text-accent-text` don't exist as project tokens). It maps them to the **AI-accent** tokens (`bg-ai`/`text-ai-text`, indigo-purple `#6457f9`/`#8b82ff`) because this is the AI Workspace — leaving the **brand/cyan** tint (`bg-accent-dim` + `border-brand/50`) for user messages produces a deliberate user↔assistant color split (user = brand, assistant/AI = ai). Active-tab styling is forced with Tailwind `!` important flags (`data-active:bg-ai! data-active:text-ai-text!`) because the shadcn `TabsTrigger` ships hard-coded `data-active:bg-background`/`dark:data-active:bg-input/30` defaults that would otherwise win in dark mode. The sidebar stays a single parent-controlled component (open/close owned by `WorkspaceShell`, unchanged) and is intentionally UI-only — sending a message appends the user bubble plus a **static** placeholder assistant reply (no async/generation), and the Specs tab's generate/download are inert placeholders, per the spec's "no backend / no AI / no Liveblocks yet" scope.

- Canvas persistence splits metadata from content per the storage model: Prisma's `Project.canvasJsonPath` holds only the Vercel Blob **URL**, while the actual `{ nodes, edges }` JSON lives in Blob at the stable path `canvas/{projectId}.json` (`addRandomSuffix: false` + `allowOverwrite: true` so each autosave overwrites the same object rather than accumulating snapshots). The autosave hook (`useCanvasAutosave`) is the single write path — debounced (1s), with volatile view fields (`selected`/`dragging`/`measured`) stripped before both content-comparison and upload so selection/hover/measure churn never triggers a save, and the **first render baselined** so the just-loaded state isn't re-written. Save status (`idle`/`saving`/`saved`/`error`) is produced inside the Liveblocks room (in `CanvasInner`) and **lifted** to `WorkspaceShell` via an `onSaveStatusChange` callback because the navbar's Save indicator lives outside the `RoomProvider`; `projectId` is just `roomId` threaded down. On-open loading runs a **one-shot** mount effect (ref-guarded) that **only loads into an empty room** — if the room already has nodes/edges (active collaboration), the load is skipped entirely so live work is never overwritten; loaded content flows through the same `onNodesChange`/`onEdgesChange` `add`-change path as templates. Save/load routes allow **owner or collaborator** (`getAccessibleProject`), consistent with the collaboration model, unlike the owner-only rename/delete/invite routes. The Vercel Blob store is **private**, so `put` uses `access: "private"` and reads go through the SDK's `get(..., { access: "private" })` (token-authenticated) rather than fetching the blob URL directly — a public `fetch` of a private blob URL fails, and using `access: "public"` on a private store throws `Cannot use public access on a private store`.

- Presence is split into two canvas-scoped overlays rather than the built-in `<Cursors />` from `@liveblocks/react-flow` (which the earlier "Next Up" note had assumed): `PresenceAvatars` (a plain overlay reading `useOthers`/Clerk `useUser`) and `CanvasCursors` (custom cursor rendering the spec asks for — colored pointer + name badge matched to each participant's presence color). Cursors render through React Flow's `<ViewportPortal>` in **flow coordinates** with an inverse-`zoom` scale, so no manual screen↔flow conversion or container-rect math is needed (the initial `flowToScreenPosition` + `containerRef.current` approach was dropped because reading a ref during render violates the `react-hooks/refs` lint rule). Cursor position is stored in flow coords (not screen coords) so every viewer sees cursors anchored to the same canvas point across independent pan/zoom. The current user is filtered out of the avatar group by matching the Liveblocks `UserMeta.id` (Clerk user ID) against `useUser().user?.id`; `useOthers` already excludes self for cursors. No `resolveUsers` is used — `UserMeta.info` is populated at auth time by `prepareSession({ userInfo })`, so `.info` is available directly on each `other`. `Presence` is `{ cursor, thinking }` (renamed from `isThinking` to satisfy this spec and the later `24-ai-presence-state` spec).
